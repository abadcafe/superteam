# 代码质量约束

## 一、代码风格规范
1. 若项目已存在代码风格规范，遵循现有代码风格
2. 若是新项目，遵循该项目所用语言标准库的代码风格
3. 命名清晰有意义
4. 函数长度合理（<80行）
5. 避免重复代码
6. 注释充分且准确
7. 代码结构清晰
8. **所有 Python 代码必须有完整的 Type Annotation**

## 二、实现原则

### 1. 严格遵循设计
- 代码实现架构文档所有功能点
- 代码逻辑必须完全符合架构文档
- 接口签名与架构文档一致
- 类型定义与架构文档一致

### 2. 最小改动原则
- 只修改必要的代码
- 不做无关的重构
- 不修改未涉及的模块

### 3. 可测试性
- 代码结构便于单测
- 依赖注入，便于 Mock
- 关键路径有日志输出
- 代码易于理解
- 代码易于扩展
- 代码易于复用
- 依赖合理

### 4. 错误处理
- 处理所有可能的错误
- 错误信息有足够上下文
- 无资源泄漏风险
- 错误正确传播

### 5. 安全性
- 无注入漏洞风险
- 无权限控制问题
- 敏感数据安全处理
- 输入充分验证

### 6. 性能
- 无明显性能问题
- 无不必要计算
- 资源使用合理
- 无内存泄漏风险

## 三、单元测试规范

### 覆盖率要求
- 所有公开函数/方法必须有测试
- 所有分支路径必须覆盖
- 所有错误处理路径必须覆盖
- 所有单测必须执行通过

### 结构规范
- 遵循当前项目已存在的单测结构
- 如果不存在，则使用该语言通行的单测结构

### 编写原则
1. 每个单测用例只验证一个行为
2. 使用 AAA 模式（Arrange-Act-Assert）
3. Mock 所有外部依赖
4. 测试名称清晰描述单测场景
5. 避免单测间的依赖
6. 测试有效且独立可重复
7. Mock 全面且合理
